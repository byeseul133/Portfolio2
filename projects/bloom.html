Bloom: Personal Task Board

Built twice. V1 went straight from idea to code: no research, no UX spec, no defined scope. V2 ran the full PDLC pipeline: brainstorm, brief, PRD, research, UX spec, then prototype. Same product. Completely different result.

What actually changes when you run the same product through a real process?

Bloom V1 was built on instinct. No research, no UX spec, no scope defined upfront. Just an idea and a code editor. V2 ran every phase of the PDLC pipeline. Same product. Completely different result.

Origin

Bloom started as something I built for myself. I wanted a personal task board that lived in one HTML file, opened instantly, and kept everything in the browser with no accounts or syncing. It worked well enough that I kept using it. When the PDLC pipeline was ready, I needed a real product to test it on. V1 was the obvious choice. So V2 wasn't a redesign for its own sake — it was the same product rebuilt from scratch through every pipeline phase.

The pipeline didn't add more work. It moved decisions earlier.

What shifted

In V1, questions came up mid-build. What should the palette be? Which features get cut? What does completing a task feel like? In V2, those questions were answered before a single line of code was written. By the time I opened a code editor for V2, the decisions that had cost me the most time in V1 were already made.

What it actually changed

Retention research

Today became the front door, not a filter

73% of people stop using a productivity tool within 30 days. The top reason: no clear starting point each morning.

In V1, Today was a filter pill. V2 makes Today the default screen — the first thing you see when you open Bloom. That came directly from a research finding, not from using the app and noticing something felt off.

UX research phase

The visual direction was decided before any code was written

Warm off-white (#F5F3EF) / deep slate blue (#2D4A6B) / Inter throughout. The feeling had a name: quiet authority.

V1's aesthetic landed through iteration. The UX research phase in V2 produced a complete spec before anything was built. When I opened a code editor to write the prototype, the aesthetic was already done.

Competitive analysis

Research cut an entire feature I thought was useful

None of the major personal task apps have a separate links section. Links belong with the task they relate to.

V1 had a dedicated Links tab. Research looked at every major personal task app and found zero of them do this. So V2 replaced the whole Links manager with a URL field on each task card. I wouldn't have cut it without the research phase.

Drop-off pattern research

Problems that were invisible in V1 got built into V2

Overdue task accumulation is the top reason people stop using task apps. Not bad design. Just tasks building up until the app feels like a guilt trip.

V1 had no safeguards for when tasks piled up. V2 has three things V1 didn't: an overdue indicator, a soft WIP limit, and a nudge when you haven't backed up in over a week.
