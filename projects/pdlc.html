The Premise

You already know how this goes.

An idea lands. It feels right. You start building. Three weeks in, the scope has grown, the problem has shifted, and the thing you're building isn't quite what you meant to build. You finish it anyway. Or you don't.

Either way, you've learned what you should have known at the start. This pipeline moves that learning to before the first line of code. It starts from wherever you are: a clear idea, or nothing yet. It ends with a clickable prototype grounded in research, defined requirements, and a resolved UX spec. Everything in between is documented. Every stage is gated.

01
Adversarial analyst Research

Validates the idea before anything is planned. The question is not what the product should be — it is whether this idea is worth building at all. Four research tiers: direct competitors, community voice, industry signals, emerging patterns. Output is a compressed verdict: what holds, what doesn't, and what would make this not worth building.

output/research.md
Gate 1.5 · Your call Did the idea hold up? Yes → continue to planning. No → pipeline stops here. Nothing gets planned for an idea the evidence didn't support. Route back to brainstorm with the findings in hand.
02
Vision writer PRD

Reads output/brief.md and output/research.md, then writes requirements once — informed from the start. No draft-then-revise cycle. Adds a User Snapshot: four lines synthesized from real research signals, covering context, prior attempts, what they actually want, and what would make them leave.

output/prd.md
Gate 2 · Your call Requirements approved? Yes → continue. Feedback → PRD reruns with targeted changes only, then Gate 2 again.
03
Experience designer UX Research

Defines every user flow, information architecture, component state, and visual direction — all traceable to the approved requirements. If anything contradicts the PRD, it flags a conflict and routes back. Maximum three rounds. If three rounds don't resolve it, a human decides.

output/ux_research.md
Gate 3 · Hard lock Does the full experience feel right? Yes → requirements and UX lock as the single source of truth. No scope changes, no flow changes after this point. After Gate 3: the prototype is not a test of the design. It is an execution of it.
04
Builder Prototype

Reads a compressed handoff from the locked UX spec: screens, palette, components, interactions. Generates a single HTML file — no setup, no build step. Every must-have flow is navigable end to end. Concrete enough for a real reaction, lightweight enough to throw away.

output/prototype.html
Gate 4 · Your call Does the prototype feel right? Yes → pipeline complete. Visual or flow issue → UX reruns, Gate 3 reopens. Feature or scope issue → PRD reruns, then UX, then prototype.
What Makes It Different

Four principles that hold.

01
Validate the idea before you write the plan.
Research runs on the raw idea, not on a draft requirements doc. You only write a plan for ideas that held up under evidence. If an idea fails at Gate 1.5, nothing has been built yet — so nothing has been wasted.
02
Each agent does one job with a clear handoff.
Research validates. PRD plans. UX specifies. Prototype builds. No agent updates another agent's output. Each handoff is a compressed summary, not a full document — lean enough to keep context clean, specific enough to not lose anything.
03
Nothing moves without your sign-off.
Five human approval gates sit between every phase. The pipeline never advances on its own. Autonomous work, deliberate handoffs.
04
Gate 3 is a hard lock. No exceptions.
UX conflicts are fully resolved before Gate 3 opens. Any pipeline that allows scope changes after the UX spec is approved will always find a reason to make one more change. Gate 3 ends that.
Why I Built It

More ideas than systems.

I have more ideas than I have systems to act on them. I built this pipeline so that the moment I have an idea, I have somewhere to put it. Something that takes it where it needs to go without me having to invent the process every time.
