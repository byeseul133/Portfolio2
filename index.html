Product Development Life Cycle

Most personal projects die in the gap between "I have an idea" and "I know what to build." I designed a five-phase pipeline, powered by purpose-built AI agents, that closes that gap. Each phase produces a real output. Each handoff requires human approval before moving forward.

Phase by Phase

01 Ideate
Brainstorm & Clarify

The idea gets a real first test before anything gets written down

A conversational AI agent acts as a thinking partner before any documentation happens. It runs real-time web research to surface what already exists, adapts to the specific industry or domain the idea lives in, and helps shape a vague impulse into a clear direction. The goal is convergence: one sentence, one platform, a handful of scope decisions, and at least one thing that's explicitly not being built.

  • Grounds the conversation in what's actually been built before committing to anything
  • Industry and domain knowledge kicks in automatically once the context is established
  • Ends with a structured handoff: core idea, platform, scope, what's out, open questions
  • A separate silent agent converts the handoff into a clean, structured project brief

02 Define
Brief & First Draft Product Requirements Document

Raw vision, captured before research can change it

Before any external research happens, a requirements agent writes the first version of the product requirements from the brief alone. No competitive data, no market trends, no outside influence. This draft becomes the baseline that all research is measured against. It runs a second time after research, updating only what the findings genuinely change. The refined version also adds a User Snapshot: four lines synthesized from real user research that ground every UX decision that follows.

  • Who it's for, what it does, what it doesn't do, and why it's worth building
  • Must Have features separated from Nice to Have and Not Building from the start
  • Runs twice: once before research to capture raw vision, once after to refine with evidence
  • User Snapshot added on the second run: context, prior attempts, real intent, and what would cause abandonment

03 Research
Parallel Research & Requirements Refine

Every assumption gets pressure-tested, then the requirements update only where the evidence demands it

Research runs across four tiers in parallel pairs: direct competitors and community voice first, then industry landscape and emerging signals. Each tier cross-validates the others. When research completes, a compressed handoff goes to the requirements agent: not the full report, just what actually needs to change. The requirements document is then refined surgically. If research doesn't give a clear reason to change something, it stays as written.

  • Tiers 1 and 2 run in parallel, then Tiers 3 and 4 run in parallel
  • Each tier's findings are validated against the others before anything reaches the requirements document
  • The requirements agent receives a compressed summary, not the full report. Only what should change.
  • Findings that hold up the draft become evidence that the original instinct was right

04 UX & Design
Flows, Architecture & Visual Direction

The full product experience is defined, and then locked

A UX research agent starts from a compressed handoff of the approved requirements, then defines every user flow, the information architecture, all component states, and the complete visual direction. Everything traces back to the intended feeling of the product. If anything contradicts the requirements document, a conflict is flagged and surfaced for review. The loop continues until zero conflicts remain. Only then do the requirements and UX lock together as the single source of truth for building.

  • Every user flow: happy path plus empty, error, loading, and first-time states
  • Platform-specific layout decisions: what works on mobile is not what works on desktop
  • Visual direction: palette, typography, motion, all rooted in the intended product feeling
  • Conflicts surfaced and resolved before this gate opens, never carried forward

05 Prototype
First-Pass Design Ideation

Something tangible to react to, before committing to real code

A prototype agent reads a compressed handoff from the locked UX research: screens, palette, components, and interactions in a tight summary, with the full spec available for reference. It generates a single HTML file that opens directly in the browser. No setup, no build step, no accounts. Every must-have flow is navigable end to end. Every component state is visible. This is a design ideation artifact — concrete enough to get a real reaction, lightweight enough to throw away if the direction changes.

  • All must-have features are clickable and navigable end to end
  • Every component state visible: default, empty, loading, error, success
  • Exact visual spec from Phase 04 including hex codes, fonts, and motion
  • A starting point for high-fidelity work in Figma or handoff to a developer

What makes it different

Research validates the vision, not the other way around

The draft requirements document is written before research runs. That way, findings are actually testing the idea rather than being reverse-engineered to support it. If research contradicts something, the product requirements document changes with evidence.

No phase starts until the previous one is approved

Human approval gates sit between every phase. The pipeline never advances on its own. Every handoff is a deliberate decision point, and every output has to earn sign-off before becoming input for the next stage.

UX and requirements conflicts are surfaced and resolved, not ignored

When UX research finds something the product requirements document got wrong, it flags a conflict. Both documents are updated until they agree. Gate 3 only opens when zero conflicts remain, so nothing contradictory makes it into the build.

Each agent receives only what it actually needs

Agents pass compressed handoffs to each other, not full documents. The research agent's summary of what should change in the requirements is a fraction of the full report. The prototype agent gets the palette and screen list, not the entire UX spec. Full documents exist for human review at gates. Inter-agent communication stays lean.

I have more ideas than I have systems to act on them. Every time an idea clicked, turning it into something real meant figuring out the same steps from scratch: what to research, how to write it down, what to build first, what to cut. I built this pipeline so that the moment I have an idea, I have somewhere to put it. Something that takes it where it needs to go without me having to invent the process every time.

V1: built without a process

The original Bloom went straight from idea to code. No research, no defined scope, no UX direction set upfront. The aesthetic was discovered through iteration. Features accumulated. The visual direction changed multiple times post-build. It worked — but it showed exactly where an unstructured process creates drag.

4-column Kanban Drag-and-drop Subtasks Due date tracking Month + priority filters Links manager Confetti on completion Rotating affirmations Pink princess aesthetic

V2: full PDLC pipeline

V2 ran every phase. Research surfaced that 73% of users abandon productivity tools within 30 days — and the Today view is the single habit anchor that determines whether a tool gets used daily or once. That finding alone restructured the entire product.

PDLC Phase What it produced
Phase 01: Brainstorm project_brief.md
Completed Scope locked before any work began. Platform confirmed as browser-first. One explicit constraint set upfront: no gamification. Research would decide the visual direction — nothing assumed.
Phase 02: PRD Draft prd_draft.md
Completed A pre-research baseline captured the raw vision. Features written down before any external input — which made it possible to see exactly what research changed, and what held up without it.
Phase 03: Research research.md
Completed Four research tiers across competitor analysis and user behavior psychology. Key finding: the Today view is not a filter — it is the product's daily ritual anchor. Backlog overwhelm is the top abandonment trigger. The Links tab had no precedent in personal task tools. Every finding with a direct PRD impact was documented.
Phase 04: UX Research ux_research.md
Completed Visual direction defined before a line of code. Research converged on "quiet authority" — warm off-white, deep slate blue, Inter typeface. Tone: settled, unhurried, precise. Every component specced with all five states. Completion interaction defined as tactile acknowledgment, not reward.
Phase 05: Prototype prototype.html
Completed Single HTML file built from the locked spec. Today view as default landing state. Overdue indicator, soft WIP limit, export nudge with last-backup date. No confetti — strikethrough draws across the task name, card fades to archive. The spec was written first. The prototype followed it exactly.

What the pipeline actually changed

Today view became the product's front door

In v1, Today was a filter pill. Research showed it is the single habit anchor — 73% of users abandon productivity tools within 30 days, and the Today view is what determines whether they come back. V2 loads it first, every time.

The aesthetic was decided, not discovered

V1's visual direction changed multiple times post-build. V2's UX spec locked the palette, typeface, and motion before code was written. Research landed on quiet authority: warm off-white, slate blue, Inter. Nothing was iterated into place.

The Links tab was cut by research

V1 had a dedicated Links manager tab. Research found no precedent for this in personal task tools — links belong with the tasks they relate to. V2 replaced the entire tab with a URL field on each task card, reducing surface area without losing the value.

Backlog overwhelm became a structural concern

Research identified overdue task pileup as the top abandonment trigger. V2 added an overdue indicator surfaced directly in the Today view, a soft WIP limit on In Progress, and an export nudge. V1 had none of these — the risk was invisible.
